# Getting Started with Seihou

Seihou (製法) is a composable, type-safe project scaffolding system. You define reusable **modules** — directories containing a Dhall definition and template files — then run `seihou run <module>` to generate projects. Seihou resolves variables from multiple sources, compiles a generation plan, and writes files to disk, tracking everything in a manifest for incremental updates.

This guide walks you through the complete workflow: initializing Seihou, creating a module from scratch, generating a project, and inspecting the results. By the end, you will have used every major CLI command.


## Prerequisites

Seihou must be installed or built from source. If building from source:

```sh
nix develop           # enter the dev shell (if using Nix)
cabal build all       # build everything
cabal run seihou -- --help   # verify the CLI works
```

No other dependencies are required.


## Step 1: Initialize Seihou

Run `seihou init` to create the configuration directory:

```sh
seihou init
```

Expected output:

```
Initialized Seihou configuration at ~/.config/seihou/

  Created: config.dhall (global defaults)
  Created: modules/ (user modules)
  Created: installed/ (git-installed modules)
```

This creates three things inside `~/.config/seihou/`:

- **config.dhall** — Global configuration file where you can set default variable values that apply to all projects.
- **modules/** — Directory for your personal modules. Place module directories here and they will be available everywhere.
- **installed/** — Directory for modules installed from git repositories via `seihou install`.

Running `seihou init` again is safe — it skips files that already exist.


## Step 2: Create your first module

A Seihou module is a directory containing a `module.dhall` file and a `files/` subdirectory with source artifacts (templates, static files, etc.). Use `seihou new-module` to scaffold one:

```sh
seihou new-module my-haskell
```

This creates the following structure:

```
my-haskell/
├── module.dhall
├── schema/
│   └── Module.dhall
└── files/
    └── README.md.tpl
```

Open `my-haskell/module.dhall` to see the boilerplate:

```dhall
{ name = "my-haskell"
, description = Some "A new seihou module"
, vars =
  [ { name = "project.name"
    , type = "text"
    , default = None Text
    , description = Some "The name of the project"
    , required = True
    , validation = None Text
    }
  ]
, exports = [] : List { var : Text, alias : Optional Text }
, prompts =
  [ { var = "project.name"
    , text = "What is your project name?"
    , when = None Text
    , choices = None (List Text)
    }
  ]
, steps =
  [ { strategy = "template"
    , src = "README.md.tpl"
    , dest = "README.md"
    , when = None Text
    , patch = None Text
    }
  ]
, commands = [] : List { run : Text, workDir : Optional Text, when : Optional Text }
, dependencies = [] : List { module : Text, vars : List { name : Text, value : Text } }
}
```

Each field:

| Field | Purpose |
|-------|---------|
| `name` | Module identifier (lowercase, hyphens allowed) |
| `description` | Human-readable description shown by `seihou list` |
| `vars` | Variable declarations with types, defaults, and validation |
| `exports` | Variables shared with dependent modules |
| `prompts` | Interactive prompts for missing variable values |
| `steps` | File generation operations (copy, template, dhall-text, structured) |
| `commands` | Shell commands to run after generation |
| `dependencies` | Other modules this module depends on |

And `files/README.md.tpl` contains a simple template:

```
# {{project.name}}

A project generated by seihou.
```

The `{{project.name}}` syntax is a placeholder — Seihou replaces it with the resolved value of the `project.name` variable during generation.


## Step 3: Customize the module

Let's make this module more useful. Replace the contents of `my-haskell/module.dhall` with:

```dhall
{ name = "my-haskell"
, description = Some "A Haskell project template"
, vars =
  [ { name = "project.name"
    , type = "text"
    , default = None Text
    , description = Some "Name of the project"
    , required = True
    , validation = Some "[a-z][a-z0-9-]*"
    }
  , { name = "project.version"
    , type = "text"
    , default = Some "0.1.0.0"
    , description = Some "Initial version"
    , required = False
    , validation = None Text
    }
  , { name = "license"
    , type = "text"
    , default = Some "MIT"
    , description = Some "License type"
    , required = False
    , validation = None Text
    }
  ]
, exports = [ { var = "project.name", alias = None Text } ]
, prompts =
  [ { var = "project.name"
    , text = "What is your project name?"
    , when = None Text
    , choices = None (List Text)
    }
  ]
, steps =
  [ { strategy = "template"
    , src = "README.md.tpl"
    , dest = "README.md"
    , when = None Text
    , patch = None Text
    }
  , { strategy = "template"
    , src = "package.cabal.tpl"
    , dest = "{{project.name}}.cabal"
    , when = None Text
    , patch = None Text
    }
  , { strategy = "copy"
    , src = "LICENSE"
    , dest = "LICENSE"
    , when = Some "IsSet license"
    , patch = None Text
    }
  ]
, commands = [] : List { run : Text, workDir : Optional Text, when : Optional Text }
, dependencies = [] : List { module : Text, vars : List { name : Text, value : Text } }
}
```

What changed:

- Added `project.version` (with a default of `"0.1.0.0"`) and `license` (with a default of `"MIT"`) variables.
- Added `validation` on `project.name` — it must match `[a-z][a-z0-9-]*`.
- Added an `exports` entry so that `project.name` is visible to any module that depends on this one.
- Added a template step for a `.cabal` file. Notice the destination uses `{{project.name}}.cabal` — placeholders work in destination paths too.
- Added a conditional copy step for `LICENSE`. The `when = Some "IsSet license"` means this step only runs if the `license` variable has a value.

Now create the template files. Replace `my-haskell/files/README.md.tpl` with:

```
# {{project.name}}

Version: {{project.version}}

A Haskell project generated by Seihou.
```

Create `my-haskell/files/package.cabal.tpl`:

```
cabal-version: 3.0
name: {{project.name}}
version: {{project.version}}
license: {{license}}
build-type: Simple

library
  default-language: GHC2024
  hs-source-dirs: src
  exposed-modules: Lib
  build-depends: base >=4.18 && <5
```

Create `my-haskell/files/LICENSE` with any license text you like (this file is copied verbatim).


## Step 4: Validate the module

Before using a module, validate it:

```sh
seihou validate-module ./my-haskell
```

If everything is correct, you will see a report with check marks for each validation:

```
Module validation: my-haskell

  ✓ module.dhall evaluates
  ✓ module name is valid
  ✓ variable names are unique
  ✓ prompts reference declared variables
  ✓ step source files exist
  ✓ exports reference declared variables
  ...

Result: valid
```

If something is wrong — say you referenced a source file that doesn't exist — the validator catches it:

```
  ✗ step source files exist
    missing: files/missing-file.tpl
```

You can also add `--lint` for advisory warnings about best practices.


## Step 5: Preview the generation plan

Before writing any files, preview what Seihou will do:

```sh
seihou run my-haskell --dry-run --var project.name=demo-app
```

The `--dry-run` flag shows the generation plan without executing it. You will see the list of operations (which files will be created, which strategy each uses, and what variable values were resolved). This is useful for verifying your module before committing to file generation.

You can also use `--diff` instead of `--dry-run` to see how the planned files compare against what already exists on disk.


## Step 6: Generate the project

Create an empty directory and generate the project:

```sh
mkdir demo-app && cd demo-app
seihou run my-haskell --var project.name=demo-app
```

If you omit `--var project.name=demo-app`, Seihou will prompt you interactively:

```
What is your project name? demo-app
```

After generation, you should see the created files:

```
demo-app/
├── .seihou/
│   └── manifest.json
├── README.md
├── demo-app.cabal
└── LICENSE
```

Check the generated `README.md`:

```
# demo-app

Version: 0.1.0.0

A Haskell project generated by Seihou.
```

The `{{project.name}}` and `{{project.version}}` placeholders were replaced with their resolved values. The `.cabal` file name itself (`demo-app.cabal`) was derived from the `{{project.name}}` placeholder in the destination path.

The `.seihou/manifest.json` file tracks what was generated — which modules were applied, what variable values were used, and a content hash for each file. This enables incremental updates on subsequent runs.


## Step 7: Check status

From inside the generated project directory:

```sh
seihou status
```

This reads `.seihou/manifest.json` and shows what was generated:

```
Seihou Status:

Applied modules:
  my-haskell    (applied 2026-03-06)

Tracked files: 3
  README.md           my-haskell   unchanged
  demo-app.cabal      my-haskell   unchanged
  LICENSE             my-haskell   unchanged

Variables: 3 resolved
```

Each tracked file shows its owning module and current state: `unchanged`, `modified`, or `deleted`.


## Step 8: Detect changes

Edit a generated file — for example, add a line to `README.md`:

```sh
echo "\n## Development\n\nTODO" >> README.md
```

Now run `seihou diff`:

```sh
seihou diff
```

Seihou compares each tracked file's current content hash against the hash stored in the manifest:

```
Changes since last generation:

  [modified]  README.md  (my-haskell)

1 modified, 0 deleted (3 tracked)
```

This tells you which generated files have been manually edited since the last run.


## Step 9: Configure defaults

Instead of passing `--var` flags every time, you can store default values in configuration files. Seihou supports three scopes:

| Scope | File | Flag |
|-------|------|------|
| Local (project) | `.seihou/config.dhall` | *(default)* |
| Namespace | `~/.config/seihou/namespaces/<ns>/config.dhall` | `--namespace NS` |
| Context | `~/.config/seihou/contexts/<ctx>/config.dhall` | `--context CTX` |
| Global | `~/.config/seihou/config.dhall` | `--global` |

Set a global default:

```sh
seihou config set license BSD-3-Clause --global
```

View all global config values:

```sh
seihou config list --global
```

Now use `seihou vars` with `--explain` to see where each value comes from:

```sh
seihou vars my-haskell --explain --var project.name=demo-app
```

The `--explain` flag shows the provenance of each resolved variable — whether it came from a CLI flag, environment variable, config file, or module default. This is invaluable for debugging variable resolution in complex setups.


## Step 10: List available modules

To see all modules Seihou can find across its search paths:

```sh
seihou list
```

Seihou searches three locations in order:

1. **Project-local**: `.seihou/modules/` in the current directory
2. **User modules**: `~/.config/seihou/modules/`
3. **Installed modules**: `~/.config/seihou/installed/`

Output shows each module with its name, description, and source:

```
Available modules:

  my-haskell   A Haskell project template   (user)

1 module found (3 sources searched)
```

To make your module available everywhere, move it to `~/.config/seihou/modules/`:

```sh
mv my-haskell ~/.config/seihou/modules/
```


## Step 11: Install modules from git

You can install modules directly from git repositories:

```sh
seihou install https://github.com/user/haskell-nix-module.git
```

This clones the repository, validates its `module.dhall`, and copies it to `~/.config/seihou/installed/<name>/`. Use `--name` to override the installed module name.

Repositories can also provide multiple modules via a **registry**. Use `seihou browse` to preview what's available, and `--module` or `--all` with `seihou install` to select which modules to install. See the [Registries and Multi-Module Repositories](registries-and-multi-module-repos.md) guide for details.


## Other commands

### `seihou run` with composition

Compose multiple modules together using the `-m` flag:

```sh
seihou run haskell-base -m nix-flake --var project.name=my-app
```

Seihou resolves dependencies automatically via topological sort. If `nix-flake` depends on `nix-base`, all three modules are loaded and executed in the correct order.

### `seihou run` flags

| Flag | Effect |
|------|--------|
| `--dry-run` | Show plan, do not write files |
| `--diff` | Show diff against current disk state |
| `--force` | Auto-resolve conflicts (accept new files) |
| `--no-commands` | Skip shell command steps |
| `--namespace NS` | Override namespace for config lookup |
| `-v, --verbose` | Show detailed progress messages |

### Re-running on an existing project

When you run `seihou run` again in a project that already has a manifest, Seihou performs a three-state diff:

- **New files** — in the plan but not in the manifest (created fresh)
- **Unchanged files** — same hash in plan and manifest (skipped)
- **Modified files** — different hash in plan vs manifest, but user hasn't edited them (updated)
- **Conflicts** — the plan wants to change a file that the user has also edited since the last generation

Conflicts are shown interactively, and you choose per-file whether to accept the new version, keep your changes, or skip the file. Use `--force` to auto-accept all new versions.


### Check for updates

See if any installed modules have newer versions available:

```sh
seihou outdated
```

This compares installed module versions against their source repositories. Use `--json` for machine-readable output.

### Upgrade installed modules

Upgrade all installed modules to the latest versions:

```sh
seihou upgrade
```

Or upgrade specific modules:

```sh
seihou upgrade haskell-base nix-flake
```

Use `--dry-run` to preview what would change without making modifications.

### Upgrade module schemas

If you have modules written for an older version of Seihou, upgrade them to the current schema:

```sh
seihou schema-upgrade ./my-module
```

Use `--dry-run` to preview changes, or `--all` to upgrade all discovered modules at once. See the [Module Authoring Reference](module-authoring.md#upgrading-modules-to-the-current-schema) for details.

### Shell completions

Generate shell completions for your preferred shell:

```sh
seihou completions bash > ~/.local/share/bash-completion/completions/seihou
seihou completions zsh > ~/.zsh/completions/_seihou
seihou completions fish > ~/.config/fish/completions/seihou.fish
```

### Help topics

Get detailed guidance on specific features:

```sh
seihou help              # list available topics
seihou help modules      # learn about module authoring
seihou help variables    # learn about variable resolution
seihou help contexts     # learn about context-based config
```
