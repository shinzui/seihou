# Add `seihou remove` Command

This ExecPlan is a living document. The sections Progress, Surprises & Discoveries,
Decision Log, and Outcomes & Retrospective must be kept up to date as work proceeds.

This document is maintained in accordance with `.claude/skills/exec-plan/PLANS.md`.


## Purpose / Big Picture

After this change, a user who has run `seihou run haskell-base` can later run `seihou remove haskell-base` to undo that module's effects: deleting the files it generated and removing it from the manifest. This is the inverse of `run`. Not every module supports removal — a module must opt in by declaring `removable = True` in its `module.dhall`. Modules that use the `Structured` strategy or that run irreversible shell commands default to non-removable because their effects cannot be cleanly reversed.

After implementation, a user can:

1. Run `seihou remove haskell-base` to delete all files generated by the `haskell-base` module and remove it from the manifest.
2. Run `seihou remove haskell-base --dry-run` to preview what would be deleted without touching the filesystem.
3. See clear warnings when a file has been modified by the user since generation (conflict), with interactive prompts to keep or delete.
4. See an error if the module was not declared as removable, or if other applied modules depend on it.

## Surprises & Discoveries

- The `Filesystem` effect's `RemoveDirectoryIfEmpty` in the pure interpreter checks both files and directories with the path prefix, preventing removal of directories that still have nested content.

- Dhall test fixture files needed `removable = False` added explicitly since they use inline records rather than schema defaults. The schema `default` record provides backwards compatibility for modules using `S.Module::{ ... }` syntax.

- The `AppliedModule` FromJSON instance uses `.:? "removable" .!= False` for backwards compatibility with manifests created before this feature was added.


## Decision Log

- Decision: Module-level `removable` flag rather than step-level granularity.
  Rationale: A module is the unit of composition and the unit users think about. Step-level removability would complicate the UX ("your module is 60% removable") without clear benefit. A module author knows whether their module's effects are reversible as a whole. If even one step is irreversible (e.g., a Structured merge or a destructive command), the whole module should be marked non-removable.
  Date: 2026-03-21

- Decision: `removable` defaults to `False` in Dhall schema.
  Rationale: Safety by default. Modules with shell commands, Structured merges, or patch operations could leave the project in an inconsistent state if their files are deleted. Authors must consciously opt in after confirming their module's effects are reversible.
  Date: 2026-03-21

- Decision: Removal operates on the manifest, not by re-loading the module definition.
  Rationale: The manifest already records which files each module generated (via `FileRecord.moduleName`). The module source may no longer be available (e.g., uninstalled). Removal should work from manifest state alone, requiring only the `removable` flag to be stored in `AppliedModule`.
  Date: 2026-03-21

- Decision: Refuse removal when other applied modules depend on the target.
  Rationale: If module B depends on module A and both are applied, removing A could leave B's generated files referencing missing variables or templates from A. The user must remove dependents first. Dependency information is available from the manifest's `AppliedModule` list cross-referenced with module definitions.
  Date: 2026-03-21

## Context and Orientation

Seihou is a composable project scaffolding system. Users run `seihou run <module>` to generate files from module definitions. The system tracks what it generated in `.seihou/manifest.json`, enabling incremental re-runs and conflict detection.

The key files and concepts involved:

**Module schema** (Dhall, in external repo `seihou-schema`): The Dhall type at `seihou-schema/Module.dhall` defines the shape of every `module.dhall` file. It currently has nine fields: `name`, `version`, `description`, `vars`, `exports`, `prompts`, `steps`, `commands`, `dependencies`. We will add a tenth: `removable`.

**Haskell domain types** at `seihou-core/src/Seihou/Core/Types.hs`: The `Module` record (line 178) mirrors the Dhall schema. `Manifest` (line 266) tracks applied modules and generated files. `AppliedModule` (line 276) records each module that was run. `FileRecord` (line 284) stores the hash, owning module, strategy, and timestamp for each generated file.

**Dhall decoder** at `seihou-core/src/Seihou/Dhall/Eval.hs`: `moduleDecoder` (line 103) maps Dhall fields to the Haskell `Module` type using `field` combinators.

**CLI commands** at `seihou-cli/src/Seihou/CLI/Commands.hs`: The `Command` ADT (line 36) and `commandParser` (line 205) define all available subcommands. Each command has an options type and a parser.

**Run handler** at `seihou-cli/src/Seihou/CLI/Run.hs`: `handleRun` implements the six-stage pipeline: module resolution, composition loading, variable resolution, plan compilation, three-state diff, execution with manifest update. The remove handler will be much simpler since it only needs to read the manifest, compute what to delete, and execute deletions.

**Filesystem effect** at `seihou-core/src/Seihou/Effect/Filesystem.hs`: Defines operations like `ReadFileText`, `WriteFileText`, `CopyFile`, `CreateDirectoryIfMissing`, `DoesFileExist`. Currently lacks `RemoveFile` and `RemoveDirectory`, which we need.

**Three-state diff** at `seihou-core/src/Seihou/Engine/Diff.hs`: Already classifies "orphaned" files (in manifest but not in plan). The removal engine can reuse this concept — all files owned by the removed module become candidates for deletion.

**Generation strategies**: `Copy` and `Template` produce standalone files that can be safely deleted. `DhallText` also produces standalone files. `Structured` merges content from multiple modules into a single file — removing one module's contribution requires re-generating from the remaining modules, which is complex and not supported in this plan. `PatchOp` steps modify files created by other modules and are similarly non-reversible in isolation.

A "removable module" is one whose effects consist entirely of creating independent files (Copy, Template, or DhallText strategies, no patch operations, and no destructive shell commands). The module author signals this by setting `removable = True`.

### Milestone 3: Core Removal Engine

At the end of this milestone, `seihou-core` exposes a `Seihou.Engine.Remove` module that computes a removal plan and executes it. The removal plan classifies each file owned by the target module as safe-to-delete (disk hash matches manifest hash), conflicted (user modified), or already-gone (deleted by user). Execution deletes safe files, prompts for conflicted files, and updates the manifest.

Create `seihou-core/src/Seihou/Engine/Remove.hs` with these exports:

    data RemovalFile
      = RemovalSafe FilePath          -- disk hash = manifest hash, safe to delete
      | RemovalConflict FilePath      -- user modified, needs confirmation
      | RemovalGone FilePath          -- already deleted from disk

    data RemovalPlan = RemovalPlan
      { targetModule :: ModuleName
      , files :: [RemovalFile]
      }

    data RemovalError
      = ModuleNotApplied ModuleName
      | ModuleNotRemovable ModuleName
      | DependentsExist ModuleName [ModuleName]

    computeRemovalPlan
      :: (Filesystem :> es)
      => Manifest
      -> ModuleName
      -> Eff es (Either RemovalError RemovalPlan)

    executeRemoval
      :: (Filesystem :> es)
      => Manifest
      -> RemovalPlan
      -> Set FilePath       -- files user chose to keep (from conflict resolution)
      -> UTCTime
      -> Eff es Manifest    -- updated manifest with module and its files removed

The `computeRemovalPlan` function:

1. Checks that the module appears in `manifest.modules`. If not, returns `ModuleNotApplied`.
2. Checks that the module's `AppliedModule.removable` is `True`. If not, returns `ModuleNotRemovable`.
3. Checks that no other applied module depends on it. This requires loading module definitions to check dependency lists, or we can store dependency info in the manifest. For simplicity in v1, we skip the dependency check and document it as a known limitation — the user is responsible for removing dependents first. We can add this check in a follow-up.
4. Filters `manifest.files` for entries where `FileRecord.moduleName` matches the target.
5. For each matching file, reads the disk content (if it exists), hashes it, and compares to `FileRecord.hash`. Classifies as `RemovalSafe`, `RemovalConflict`, or `RemovalGone`.

The `executeRemoval` function:

1. For each `RemovalSafe` file, calls `removeFile`.
2. For each `RemovalConflict` file not in the keep-set, calls `removeFile`.
3. Skips `RemovalGone` files.
4. After deletions, walks parent directories bottom-up and removes empty ones via `removeDirectoryIfEmpty`.
5. Returns a new `Manifest` with the module removed from `modules` and its files removed from `files`.

Acceptance: Unit tests (Milestone 5) pass. `cabal build all` succeeds.


### Milestone 4: CLI Integration

At the end of this milestone, `seihou remove <module>` is a working CLI command. It reads the manifest, computes a removal plan, shows a preview, prompts for confirmation, executes deletions, and updates the manifest.

**Command type** in `seihou-cli/src/Seihou/CLI/Commands.hs`: Add `Remove RemoveOpts` to the `Command` ADT. Define:

    data RemoveOpts = RemoveOpts
      { removeModule :: ModuleName
      , removeDryRun :: Bool
      , removeForce :: Bool
      , removeVerbose :: Bool
      }

Add `command "remove" removeInfo` to `commandParser`'s subparser list. Write `removeInfo`, `removeParser` following the pattern of the existing `run` command.

**Handler** — create `seihou-cli/src/Seihou/CLI/Remove.hs`:

    handleRemove :: RemoveOpts -> IO ()

The handler:

1. Reads the manifest from `.seihou/manifest.json`. If missing, prints "no manifest found" and exits 1.
2. Calls `computeRemovalPlan manifest moduleName`.
3. On error (`ModuleNotApplied`, `ModuleNotRemovable`, `DependentsExist`), prints a user-friendly message and exits 1.
4. Displays the removal plan: lists files to delete (safe), files with conflicts, and files already gone.
5. If `--dry-run`, prints the plan and exits 0.
6. If conflicts exist and `--force` is not set, prompts the user for each conflicted file (keep/delete/abort).
7. Executes the removal plan.
8. Writes the updated manifest.
9. Prints a summary: "Removed module <name>. Deleted N files."

**Main dispatch** in `seihou-cli/src/Main.hs` (or wherever commands are dispatched): Add a case for `Remove opts -> handleRemove opts`.

## Idempotence and Recovery

Running `seihou remove <module>` when the module is not in the manifest prints an error and exits 1 — it does not crash or corrupt state. Running it after a successful removal is idempotent (the module is simply not found in the manifest).

If removal is interrupted partway (e.g., user kills the process), the manifest is only written at the end, so the state remains as if removal never happened. The user can re-run the command. Files that were already deleted will be classified as `RemovalGone` and skipped.

The Dhall schema change is backwards-compatible — existing modules without `removable` get `False` via the default. Existing manifests without `removable` on `AppliedModule` decode with `False` via `.:? "removable" .!= False`.
