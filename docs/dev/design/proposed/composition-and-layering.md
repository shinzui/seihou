# Composition and Layering

| Field | Value |
|---|---|
| **Status** | Proposed |
| **Created** | 2026-03-01 |
| **Subsystem** | Core вҖ” Composition |

## Overview

Seihou modules compose via explicit layering. Modules declare dependencies, and the system determines execution order through topological sort of the dependency graph. When multiple modules target the same output file, strategy-dependent patch semantics determine how their contributions are merged.

## Motivation

Implicit module stacking (where composition is determined by order on the command line) leads to fragile, order-dependent behavior. Explicit dependency declaration makes composition predictable and debuggable: the system knows exactly which modules contribute to each file and in what order.

## Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Composition model | Explicit layering with declared dependencies | Most correct; avoids implicit ordering bugs |
| Execution order | Topological sort of dependency graph | Dependencies always run before dependents |
| Structured file merge | Dhall record merge (`/\`) | Leverages Dhall's native deep merge semantics |
| Text file patching | Declarative operations (append, replace, prepend) | Predictable, composable, auditable |
| Copy file conflict | Last-writer-wins with warning | Simple; user can override via --force or config |
| Conflict handling | Show diff, user decides per-file | Fits plan-first philosophy |

## Dependency Graph

### Declaration

Modules declare dependencies in their `module.dhall`:

```dhall
{ name = "haskell-with-nix"
, dependencies = ["haskell-base", "nix-flake"]
, ...
}
```

### Resolution

```
haskell-with-nix
       в”Ӯ
       в”ңв”Җв”Җв–¶ haskell-base
       в”Ӯ
       в””в”Җв”Җв–¶ nix-flake
                в”Ӯ
                в””в”Җв”Җв–¶ nix-base
```

The system builds a DAG and performs topological sort to determine execution order:

```
nix-base вҶ’ nix-flake вҶ’ haskell-base вҶ’ haskell-with-nix
```

### Domain Model

```haskell
data CompositionGraph = CompositionGraph
  { nodes :: Map ModuleName Module
  , edges :: Map ModuleName [ModuleName]  -- module вҶ’ its dependencies
  }
  deriving stock (Eq, Show, Generic)

-- | Result of topological sort
data ExecutionOrder
  = ExecutionOrder [ModuleName]           -- Valid order
  | CyclicDependency [ModuleName]         -- Cycle detected
  deriving stock (Eq, Show, Generic)

-- | Resolve execution order from a composition graph
resolveOrder :: CompositionGraph -> ExecutionOrder
```

## Patch Semantics by Strategy

When multiple modules target the same output file, the system applies strategy-dependent patch operations.

### Structured Files (JSON/YAML) вҖ” Dhall Record Merge

When two modules generate the same structured file, their Dhall records are merged using Dhall's right-biased record merge (`//`) for top-level fields and deep merge (`/\`) for nested records.

```dhall
-- Module A generates:
{ name = "my-project"
, dependencies = { base = ">=4.16" }
}

-- Module B patches with:
{ dependencies = { effectful = ">=2.0" }
, extra = { ci = True }
}

-- Result (A /\ B):
{ name = "my-project"
, dependencies = { base = ">=4.16", effectful = ">=2.0" }
, extra = { ci = True }
}
```

Merge order follows the execution order (later modules override earlier ones for conflicting scalar fields).

### Text Files (Template) вҖ” Declarative Operations

Text files support declarative patch operations:

```haskell
data PatchOp
  = AppendSection Text Text    -- section marker, content to append
  | PrependSection Text Text   -- section marker, content to prepend
  | ReplaceSection Text Text   -- section marker, replacement content
  | AppendFile Text            -- content to append at end of file
  | PrependFile Text           -- content to prepend at start of file
  deriving stock (Eq, Show, Generic)
```

Modules declare patches in their step definitions:

```dhall
{ strategy = "template"
, src = "gitignore-additions.tpl"
, dest = ".gitignore"
, patch = Some
    { op = "append-file"
    , content = None Text       -- Content comes from src file
    }
}
```

Section markers use comment-delimited boundaries:

```
# --- seihou:haskell-base ---
dist-newstyle/
*.hi
*.o
# --- /seihou:haskell-base ---

# --- seihou:nix-flake ---
result
.direnv/
# --- /seihou:nix-flake ---
```

### Copy Files вҖ” Last-Writer-Wins

When two modules copy different files to the same destination, the later module (per execution order) wins. A warning is emitted:

```
Warning: File 'LICENSE' generated by both 'haskell-base' and 'license-override'.
         Using output from 'license-override' (last in execution order).
```

## Layer Result

```haskell
data LayerResult = LayerResult
  { generatedFiles :: Map FilePath GeneratedFile
  , warnings       :: [CompositionWarning]
  , appliedPatches :: [(ModuleName, FilePath, PatchOp)]
  }
  deriving stock (Eq, Show, Generic)

data GeneratedFile = GeneratedFile
  { content        :: ByteString
  , sourceModule   :: ModuleName       -- Which module generated this
  , contributors   :: [ModuleName]     -- All modules that contributed
  , strategy       :: Strategy
  }
  deriving stock (Eq, Show, Generic)

data CompositionWarning
  = FileOverwritten FilePath ModuleName ModuleName  -- path, old writer, new writer
  | UnusedExport ModuleName VarName                 -- Module exports var nobody imports
  deriving stock (Eq, Show, Generic)
```

## Variable Flow Between Modules

Variables flow between composed modules through the export/import mechanism:

1. Module A declares `var: project.name` and `exports: [{ var: "project.name" }]`
2. Module B declares `dependencies: ["module-a"]`
3. During variable resolution, Module B can reference `project.name` because Module A exports it


## Conflict Detection and Reporting

### Conflict Types

```haskell
data CompositionConflict
  = ScalarMergeConflict     -- Two modules set same scalar field in structured file
      { conflictFile   :: FilePath
      , conflictField  :: Text
      , conflictLeft   :: (ModuleName, Text)  -- (module, value)
      , conflictRight  :: (ModuleName, Text)
      }
  | SectionOverlap          -- Two modules patch same section in text file
      { conflictFile    :: FilePath
      , conflictSection :: Text
      , conflictLeft    :: ModuleName
      , conflictRight   :: ModuleName
      }
  | CopyConflict            -- Two modules copy to same destination
      { conflictFile  :: FilePath
      , conflictLeft  :: ModuleName
      , conflictRight :: ModuleName
      }
  deriving stock (Eq, Show, Generic)
```

### Resolution Strategy

For v1, conflicts are surfaced to the user during the approval phase:

1. The diff view highlights conflicting files
2. For each conflict, both versions are shown
3. The user chooses: accept left, accept right, or abort
4. `--force` auto-accepts the last-writer-wins result

## Business Rules

- Dependency declarations must form a DAG (no cycles)
- A module can depend on any number of other modules
- Execution order is deterministic for a given dependency graph
- Patch operations are applied in execution order (dependencies before dependents)
- Section markers in generated files must be preserved by users for future patch operations to work
- Modules generating the same file must use compatible strategies (e.g., two modules cannot template and copy the same file)

## Edge Cases

| Case | Behavior |
|---|---|
| Diamond dependency (AвҶ’B, AвҶ’C, BвҶ’D, CвҶ’D) | D executes once; topological sort handles this |
| Module depends on itself | Rejected as circular dependency |
| Two modules patch same section | SectionOverlap conflict surfaced to user |
| Patch target file doesn't exist yet | Error if patching; OK if first module creates it |
| Module with no files, only exports | Valid; acts as a variable-providing module |
| Three modules contribute to same structured file | Deep merge applied in execution order |
