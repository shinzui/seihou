# Project-aware module updates

## Goal

`seihou update` advances a recorded module or recipe application without first
mutating the shared installed cache and without erasing user edits. Planning is
read-only; publication occurs only after migrations, reconciliation, and
commands succeed.

## Persistent model

Manifest schema version 4 retains the legacy applied-module and flat-variable
views while adding top-level applications. An application is identified by the
requested module or recipe plus ordered explicit additional roots. Versions,
resolved values, and recipe-expanded dependency membership are deliberately
excluded so a later update replaces the same application.

Each application records ordered module instances, their parent bindings,
source/version, accepted values, and successful command receipts. Each file
records its application owners, the hash of applied disk bytes, and a reference
to the generated ancestor. The ancestor is stored at
`.seihou/baselines/<sha256>`; the manifest accepts only validated hexadecimal
references.

## Merge and reconciliation

Text reconciliation uses the previous generated ancestor as base, current disk
bytes as ours, and candidate generated bytes as theirs. Pure identity cases are
handled without a process. Dual edits use `git merge-file --stdout --diff3`.
Driver absence, failure, or binary input is conservative: it produces a
conflict, never an overwrite.

Repeated generation and patch operations are materialized into one desired
state per path before classification. Unchanged obsolete files are deleted;
edited orphans remain tracked until explicitly deleted or detached. A targeted
update is rejected if the selected applications do not include every owner of
a shared or colliding path.

## Staging and publication

Remote origins are cloned once into a session directory. Saved instance inputs
are checked against candidate declarations, migrations are deduplicated by
module source/version transition, declarative operations are applied to a
staged filesystem, and one update plan contains migration, file, and command
outcomes.

Apply rechecks candidate and project snapshots. A journal protects managed
project paths, migration directories, installed-cache artifacts, baselines,
and manifest state. The final manifest is the durable commit marker; recovery
restores backups unless that marker already matches. Cache publication and
baseline pruning therefore cannot make a failed project update appear current.

## Command receipts

A stable fingerprint covers rendered command content, working directory,
qualified module owner, and duplicate occurrence. Update defaults to executing
only fingerprints without a successful receipt. Run-all and disabled policies
remain explicit. Receipts are finalized against the complete current
declaration set only after total command success.

## Limits

Binary files have no automatic merge. Diff3 is line-oriented rather than
language-aware. Shell commands may affect external state outside the recovery
journal. Blueprints and prompts are excluded because their agent-authored
output is not a deterministic composition. Multiple named applications of the
same root and additions remain future work.
